YC创业课总结

在一个微信群看到李笑来老师的推广链接，很感兴趣就加入了这个课程学习。对于坚持完成20节课程我倒没太大压力，因为之前坚持每天写阅读记录也坚持了150多天。唯一担心的就是怕为了完成任务，不求甚解。

从目前来看，部分课程确实因为不感兴趣没仔细看，应付交了作业，但多数课程确实都值得反复看，然后在实践中摸索。

小小收获：

  1. 结识了几个新朋友，偶尔用Dove聊天讨论学习下。

  2. 学到不少关于产品、运营方面的经验教训。比如第6课、第7课，非常实用。

  3. 买了几本好书，都源于课程里的推荐。《创业维艰》《从0到1》《至关重要的关系》《成功，动机与目标》

  4. 开始习惯使用Markdown，并用Hexo搭建了一个博客 http://www.tuijiankan.com

古语说的对：“纸上得来终觉浅，绝知此事需躬行”，光有理论是不够，仍然要靠实践去融会贯通。

于是我们尝试在公司召集小伙伴们一同观看YC创业课，看完后让大家自由发表观点，结合目前工作来讨论哪些经验可以用得上。开始的时候，大家都谨慎，但是慢慢的大家都会提出自己在工作中遇到的问题，即使没有结合视频给出解决方案，但也起到了内部沟通、发现问题的作用。这种方法很值得推广。

仔细看了下其他人提交的YC创业课总结，竟然有不少人提到最大的感悟是“不想创业”了，至少让“创业冲动”冷静下来。

课程让我了解到《从0到1》作者对精益创业不看好的原因，他认为好的创见是可以预期规划的，靠产品迭代、用户反馈来做产品是思维懒惰的表现。可我又看到不少课程的演讲者都在用“精益创业”的方法来实现自己的创见。难道这不矛盾么？

直到看了群里讨论提到的一本书《成功，动机与目标》，我才发现peter的说法和多数人的做法其实都不矛盾，长远规划要用“为什么”思维去想，而对于眼前复杂的工作，用“是什么”思维才能加速启动，不落入拖延症的死亡深渊。

只谈“为什么”的人往往会吹牛B，而只谈“是什么”的人，格局又变的太小，要根据具体问题，灵活切换思维模式。要获得持久的动力，必须要思考“为什么”，而为了解决问题、迈出坚实的第一步就要思考“是什么”。

结合着我们公司的网站，“为什么”问题：要想到我们公司存在的目的在于帮助中小企业做网络营销。“是什么”就是为了达成这个目标具体要做些什么：

  • 给用户提供简单好用的广告发布平台，百度收录效果要好。

  • YY语音培训，给大家讲如何发帖容易被收录，怎样做网络营销推广。

  • QQ群，方便与用户交流沟通，也方便用户之间交流，互相学习网络营销推广。

  • 各种工具，比如SEO工具，标题生成工具等，辅助用户更好的撰写广告内容。

  • 做B2B网址导航，让大家去各个好的网站发广告。

还有关于用户调研发现了自己以前的一些错误做法，甚至错误的用户反馈调查能误导你的产品改进方向。

总之，这套课程对我的学习和工作都有帮助。

最近公司网站被黑客盯上，天天被DDos攻击，焦头烂额的忙。两天搬了两家机房，大量用户因为访问不了网站，焦急抱怨，也有很多忠实粉丝给了我们很多支持、鼓励。

跟YC课程里说的一样，创业不应该只是一时冲动，要解决用户的需求，要学会运营，要学会推广，一堆堆的问题需要创业者去实践中摸爬滚打，不要只看到别人台前的风光，要耐得住寂寞去修炼内功。

同时，创业者一定要谨慎的乐观，你的员工，你的用户，你的产品才是最重要的资产。

焦头烂额应对DDos攻击记录

五一三天假，以为和往年一样风平浪静。等到到5月2日晚上8点左右，不少用户反馈登录不上黄页88网，一开始以为又是机房出问题（一年平均出个七八次事故，烂机房！），但没多久机房代理商电话过来说“你们网站被DDos攻击，机房扛不住直接把你们网站IP封了。而且机房很愤怒，因为大规模影响了其他客户，要求你们马上把服务器搬走，IP解封也是不可能的！”

之后是各种电话通知联系，技术总监在外地，第二天才能开车赶回北京，不能马上赶到机房处理。大家都焦急的等着。

终于等到了5月3日，代理商建议搬到他自己在农大南路的机房，于是我们开了三辆车，装了20台服务器和一大堆网线、电源线就出发了。

赶到机房大概是下午2点左右，我们几个人开始把一台台服务器上架，调试网络、修改IP、域名解析，系统的各种服务启动、配置。一直到晚上11点左右，才算基本搞定。

大家都舒了口气，然后持续的观察调试，后续搞到凌晨2点多才睡。

5月4日大清早，黑客又发动了新一轮的DDos攻击，代理商机房也瘫痪了，而且IP同样被上级运营商封掉，刚打开一个子域名测试，立马就被黑客发现，发动攻击。

上级网络运营商告诉机房代理商IP需要三天后才能解封。一方面他们也很无奈，另一方面甚至有点后悔让我们网站搬到他的机房。

可对于我们这类型的商业信息发布平台，停站3天，那么百度可能就不再收录，网站基本就算死掉了。

心急如焚的我们马上一边联系新的上架机房，另一方面咨询朋友，了解各种抗DDos攻击方案：

  1. 一家知名网络招聘公司的朋友他们的经验是：用阿里云当服务器，开阿里云盾抗过去。
  2. 国内知名SEO站长的经验：把服务器放到群英硬防机房。 http://www.qy.com.cn/ddos/
  3. 其他朋友的建议：使用百度加速乐、360网站助手或百度安全宝。

最后：

机房确定了，黄页88服务器会放到小米公司在丰台区的一个机房，质量据说很不错，虽然价格不菲。

防DDos方案也确定了，直接找到安全宝的老大马杰开了个5G防护套餐。

到5月4日下午3点左右，我们一行人又开着车载着20多台服务器直奔新机房，幸好路上不堵车，我们边聊天边听着90年代的摇滚乐，还挺High。

因为这个机房很大，所以流程挺多，办上架手续也等了半天，而且这个机房带宽不是瓶颈，居然拉服务器的小推车不够用，被我们嘲笑了半天。

服务器一台台的搬上搬下，终于上架了，又是一通调试、配置，好在我们之前搬机房时候学到了经验，把每台机器、每个网口都打了标签贴上去，让这次上架变的异常顺利。

差不多到晚上10点半的时候，网站部分地区已经可以访问了，由于域名使用的万网的DNS解析，所以需要48小时才能全国生效。后续我们继续修一些由于DNS没有全面生效造成的一些访问问题，一直折腾到第二天凌晨1点多。

老大开着车载我和我们的技术总监，边聊网站后续的构架改进方案，边听《唐朝》第一张专辑，这时虽然又累又困，但很有成就感。凌晨的北京，一堆号称24小时营业的店原形毕露，只有麦当劳最靠谱，接近打烊，但是还是卖给了我们三个麦辣鸡腿堡和三杯可乐。

回到公司打印了几份网站备案用的文件，然后飙车回家，只用了7分钟。

谢天谢地，第二天我没有接到紧急电话，一觉睡到9点半，爽的很。

黄页88被DDos攻击恢复过程经验心得

机房

  • 尽量选择优质机房，最好有硬件防火墙，和大的网站平台放在一起。

  • 大的机房上架流程复杂，最好提前预约走流程，去机房记得带身份证。

  • 优质机房一般会测电，防止服务器上架短路。

  • 机房对电源、网线的布置要求美观严格，最好按他们的规范布线。（有备用电源的一左一右插电源线）

服务器

  • 服务器操作系统最好保持一致，否则维护操作复杂度提升。

  • 汽车后备箱放服务器最多放7-8台，注意缓冲防护，免得急刹、拐弯磕碰。

  • 服务器的网卡每次重装系统都可能发生变化，记得标注好网卡口，方便上架插线。

  • 每台服务器前后面都要贴上标签（以功能和IP末位组合命名），方便接显示器调试。

软件

  • 网站构架一定要松耦合，避免把鸡蛋都放到一个篮子里。

  • 网站尽量云平台化，国内推荐用阿里云，国外推荐用AWS。

  • 要注意网站安全防护，可以采用第三方专业服务，比如安全宝。（还有360网站助手、加速乐可选）

  • 网站域名解析对系统恢复影响很大，万网解析很慢，需要48小时，建议使用DNSPOD。

  • 网站搬机房，注意要迁移备案，免得被注销后重新备案更麻烦。

  • 如果网站关停时间过长，最好到百度站长平台后台设置。

用户

  • 网站打不开安抚用户，给用户一个服务恢复的大概时间。

  • 被攻击要对外发声，警告攻击者，同时给用户以信心。

  • 平时要注意增加维护各个发声渠道，比如微信、微博、QQ空间、QQ群，用来保持与用户的畅快沟通。

  • 被DDos攻击，可以找公司注册地的网警报案，需要提交被攻击时段的日志。

阿里云Centos6安装配置Nodejs、Nginx、Hexo操作记录

之前使用本地运行hexo，生成静态文件分发到Github和Gitcafe的玩法，也搭建了两个网站。 、

但是觉得每次都要使用hexo g && hexo d命令生成静态然后分发很不方便，于是重装了自己的阿里云服务器，打算直接让hexo跑起来，体验写个Markdown文件就能生成博客的快感。

怎么操作呢？

  1. 阿里云重新安装，点击更换磁盘，虽然有很可怕的提示，但是不要管。一步步按流程走，我选择安装了Centos6.5 64位操作系统。

  2. 运行Mac下的Terminal，输入 ssh root@yourdomain.com 访问。（如果以前访问过记录过rsa秘钥文件，记得删除掉，否则无法登录服务器）

  3. 进入系统后参考http://codybonney.com/installing-node-js-0-10-24-on-centos-6-4/ 一步步安装开发工具和Nodejs。

  4. 按照这个教程安装Nginx：http://codybonney.com/installing-nginx-on-centos-6-4/

  5. 使用Mac下的Sftp软件 YummyFTP上传本地hexo文件到服务器，目录随意，我使用的是 /home/hexo

  6. 因为本地是可以运行hexo的，但是传到服务器后，还需要全局安装Hexo,命令：npm install -g hexo ，然后运行hexo s 就可以通过 http://youdomain.com:4000 看到你的博客了。

  7. 但是不可能让用户通过加端口的名字去访问，所以需要设置nginx的反向代理，监听4000端口,default.conf文件内容如下。

    server {
    listen 80;
    server_name yourdomian.com, www.youdomain.com;

    location / {
        proxy_pass              http://127.0.0.1:4000/;
        proxy_redirect          off;
        proxy_set_header        X-Real-IP       $remote_addr;
        proxy_set_header        X-Forwarded-For $proxy_add_x_forwarded_for;
        }
  }

之后重启nginx。

  1. 让Hexo后台运行，命令 hexo s &,这样就不会关掉terminal网站就打不开了。

  2. 以后写博客，可以在服务器运行 hexo n "title" 来生成空的markdown文件，然后敲博客就行了。 如果本地写好，用sftp往服务器传也可以。

  3. 竣工！睡觉~

Git命令学习记录

Git学习

安装完成后的配置命令：

$ git config —global user.name “Your Name”

$ git config —global user.email “email@example.com”

小贴士：

ls -ah 显示隐藏文件夹.git命令

git diff HEAD — filename 查看工作区的版本与最新版本库版本的文件区别

git reset即可回退版本，也可以把暂存区的修改退回到工作区，用HEAD时表示最新版本。

Git跟踪的是修改而不是文件。

创建SSH的操作命令：

ssh-keygen -t rsa -C “youremail@example.com”

生成位置在用户主目录下的.ssh文件夹下，有id_rsa 和 id_ras.pub两个文件。

git remote add origin git@github.com:yougithubusername/youreponame.git 添加远程库

如果第一次往空的github推送，推送命令 git push -u origin master让本地库与远程建立关联，以后可以去掉-u参数。

查看远程分支详情命令 git remote -v,会显示远程库的完整访问地址，而不是名称.

git config --global color.ui true 可以让git显示语法高亮。

.gitignore用于忽略某些文件和文件夹，支持通配符。目录结尾斜杠不能省略，否则就是忽略文件。

Git版本恢复

git reset —hard head^ 恢复到上个版本

git reset —hard head^^ 恢复到上上个版本

git reset —hard commit_id 恢复到某个版本

git reflog 记录所有的命令历史，便于恢复到任意版本

git log —pretty=oneline 记录历史commit，方便恢复到某个commit版本。

Git的操作流

git add filename -> git commit -m “description” -> git push origin master

Git的工作方式理解（个人理解）

  • 直接能看到的文件所在区叫“工作区”。

  • ./git里的是版本库，包含了暂存区和Head版本区

  • 每次git add 是为了把工作区的文件放到暂存区跟踪版本。

  • 每次git commit 是为了把暂存区的内容放到真正的版本区里。

  • 每次git push 是为了把本地的代码推上线

Git完整操作流程测试

  1. 本地建个文件夹 justtest.

  2. 进入文件夹后输入 git init 初始化本地仓库。

  3. vi test.md 创建个文件随便输入点东西。:wq!保存退出。

  4. git add test.md

  5. git commit -m "justtest"

  6. Github建立一个空的库，比如我命名为justtest,生成的地址为：git@github.com:joeseesun/justtest.git

  7. 添加远程库 git remote add origin git@github.com:joeseesun/justtest.git

  8. 推送本地库到远程：git push -u origin master

  9. 打开github你的justtest仓库，应该就可以看到test.md文件。

Git 分支操作

  • 查看分支 git branch

  • 创建分支 git branch <branchname>

  • 切换分支 git checkout <branchname>

  • 删除分支 git branch -d <branchname>

  • 合并其他分支到当前分支 git merge <branchname>

  • 创建并切换到新分支 git checkout -b <branchname>

  • 删除远程分支 git push origin :<branchname> 原理推一个空分支上去，相对于删除，冒号不能少。

  • 强行删除未合并分支 git branch -D <branchname>

Git分支冲突合并

  • 如果新建的分支和主分支都做了修改并且commit，git merge <branchname>命令会造成文件冲突，必须修改后add和commit

  • git log --graph可以查看分支合并图,git log --pretty=oneline --graph --abbrev-commit 显示更简洁的合并图。

  • git的默认使用fast-foward分支合并模式，缺点是删除分之后会丢掉分支历史信息。使用--no-ff参数可以禁用fast-foward模式。

  • 如果修复bug，可以创建bug分支去处理，但是当前工作还没完成的情况下，可以使用git stash命令去记录当前分支的工作区。修复合并完bug分支后，在之前的工作区使用git stash pop来恢复工作区并删除快照（如果单独操作命令是 git stash apply,恢复命令是git stash drop）,查看工作区快照命令 git stash list。

Git标签

  • git tag <tagname>用于新建标签，默认为HEAD,也可以指定Commit id。

  • git tag -a <tagname> -m "description"写标签信息

  • git show <tagname> 来看具体tag的内容

  • git push origin <tagname> 推送一个标签到远程

  • git push origin --tags 推送所有标签到远程

  • git tag -d <tagname> 本地删除一个标签

  • git push origin :refs/tags/<tagname> 远程删除一个标签

Git命令配置别名

  • git config --global alias.st status 将git status简化为git st;

  • git config --global alias.ci commit 将git commit简化为git ci;

  • git config --global alias.co checkout 将git checkout简化为git co;

  • git config --global alias.br branch 将git branch简化为git br;

  • git config --global alias.last 'log -1' 显示最近一次commit信息;

  • git config --global alias.unstage 'reset HEAD' 将git reset HEAD简化为git unstage;

  • git config --global alias.lg "log --color --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit" 图形分支图着色
